============================================================
COMO USAR ESTE TXT
============================================================

Este archivo complementa endpoints_postman.txt.

Aqui se documenta por endpoint:
- Que pide: params, query, body y auth.
- Que devuelve: forma general de la respuesta.
- Validaciones: reglas que pueden cortar con 400, 401, 403, 404, 409, etc.
- Flujo: que hace internamente.
- Casos: escenarios utiles para probar en Postman.
- Prueba/hallazgo: estado de lo que ya se probo.

Headers para rutas protegidas:

Authorization: Bearer {{token}}
Content-Type: application/json

Login base:

POST {{base_url}}/auth/login
Body:
{
  "username": "admin",
  "password": "1234"
}

Respuesta:
- 200 OK
- ok=true
- token
- usuario

Diferencia entre objetos creados en PLANIFICADA y ACTIVA:
- Cambio aplicado 2026-05-04.
- Ahora la BD y la API usan el campo:
  estado_operacion_creacion
- Este campo guarda el estado real de la operacion al momento de crear el objeto.
- Valores esperados:
  - PLANIFICADA: objeto creado durante planeacion.
  - ACTIVA: objeto creado durante ejecucion activa.
- Si despues se cambia el estado de la operacion, el objeto conserva su estado_operacion_creacion original.
- El campo se llena automaticamente con trigger de BD en inserts nuevos.
- Se agrego backfill a registros existentes usando el estado actual de su operacion. Nota: para datos historicos previos al cambio no se puede reconstruir el estado exacto de creacion.
- Tablas cubiertas:
  asignacion_operacion_personal, vehiculo_operacion, operacion_equipo, uso_equipo_operacion, grupo_operacion, grupo_personal, grupo_equipo, grupo_vehiculo, mando_operacion, chat_operacion, mensaje_chat, puntos_interes, dibujo_libre_operacion, area_interes, ruta_operacion, marca_edificio, ruta_navegacion, zona_operacion, tracking_personal, tracking_vehiculo, aviso_operacion, novedad_operacion, operacion_evento.

- La operacion sigue teniendo su estado actual en operacion.estado. Ese campo no reemplaza estado_operacion_creacion; son cosas distintas.
- Las rutas tacticas tambien tienen su propio estado de ruta: PLANIFICADA, ACTIVA, COMPLETADA o CANCELADA. Ese estado es de la ruta, no necesariamente el estado de la operacion al crearla.

GET /db-test

Prueba:

POST /auth/login
Auth: no
Pide body:
{
  "username": "admin",
  "password": "1234"
}
Devuelve:
- ok=true
- token JWT valido por 8h
- usuario con id_usuario o id_personal, username, rol, nombre, apellido, puesto, tabla
Validaciones:
- 400 si faltan username o password.
- 401 si no existe usuario/personal o password no coincide.
- 403 si el usuario existe pero activo=false.
Flujo:
- Busca primero en tabla usuario.
- Si no encuentra, busca en tabla personal.
- Valida activo.
- Compara password con bcrypt.
- Actualiza ultimo_acceso.
- Firma JWT con sub, username, rol y tabla.
Casos:
- Login admin correcto.
- Password incorrecto.
- Usuario inexistente.
- Usuario inactivo.
Prueba:
- OK 200 con admin/1234.

GET /me
Auth: si
Pide:
- JWT.
Devuelve:
- id_usuario, username, rol, nombre, apellido.
Validaciones:
- 401 si no hay token valido.
- 404 si req.user.sub no existe en tabla usuario.
Flujo:
- Lee req.user.sub y consulta tabla usuario.
Caso importante:
- Si el token viene de personal, puede responder 404 porque esta ruta consulta usuario, no personal.
Prueba:
- OK 200 con admin.

GET /config/cesium-token
Auth: si
Pide:
- JWT.
Devuelve:
- ok=true
- token de Cesium.
Validaciones:
- 401 si no hay token valido.
- 404 si la variable/token esta vacia.
Flujo:
- Lee configuracion de servidor y responde el token.
Prueba:
- OK 200. No se pego el valor real porque es sensible.

- Editar PLANIFICADA: 200.
- Editar ACTIVA/CERRADA/CANCELADA: 400.
Prueba:
- OK con operacion POSTMAN_TEST planificada.
- OK flujo dashboard/asignacion con /ops/2: PUT temporal + GET /ops/2/mapa confirmo la fecha y despues se restauro.

PATCH /ops/:id/estado
Auth: si
Pide params:
- id entero.
Pide body:
{
  "estado": "ACTIVA"
}
Devuelve:
- ok=true
- operacion actualizada.
Validaciones:
- 400 si id invalido.
- 400 si estado no es PLANIFICADA, ACTIVA, CERRADA, CANCELADA.
- 404 si operacion no existe.
- 409 si la transicion no esta permitida.
Flujo:
- Usa transaccion.
- Maquina de estados:
  PLANIFICADA -> ACTIVA o CANCELADA
  ACTIVA -> CERRADA o CANCELADA
  CERRADA -> ninguna
  CANCELADA -> ninguna
- Si pasa a ACTIVA:
  - fecha_inicio=NOW()
  - crea/reactiva chat_operacion
  - agrega mensaje SISTEMA de activacion
- Si pasa a CERRADA:
  - fecha_fin=NOW()
  - agrega mensaje SISTEMA de cierre
  - desactiva chat
  - conserva asignaciones para historial
- Si pasa a CANCELADA:
  - libera personal, vehiculos y equipos
  - desactiva chat
  - agrega " - CANCELADA" al nombre
Casos:
- PLANIFICADA a ACTIVA: 200.
- ACTIVA a CERRADA: 200.
- ACTIVA a PLANIFICADA: 409.
- CERRADA a ACTIVA: 409.
Prueba:
- Se uso para preparar operacion ACTIVA id=6.

============================================================
10. CASOS GENERALES PARA POSTMAN
============================================================

Auth:
- Sin token en rutas protegidas: esperar 401.
- Token invalido o expirado: esperar 401.

IDs:
- Enviar id no numerico en params: normalmente 400.
- Enviar id inexistente: normalmente 404.

Operaciones cerradas/canceladas:
- Asignaciones de personal, grupos, vehiculos, equipos y rutas de navegacion deben bloquear modificaciones en CERRADA/CANCELADA.

Soft delete:
- POIs: DELETE activo=false, restore activo=true.
- Areas: DELETE estado=ELIMINADA, restore estado=ACTIVA.
- Edificios: DELETE estado=INACTIVO, restore estado=ACTIVO.
- Dibujos: DELETE activo=false, restore activo=true.
- Rutas tacticas: DELETE estado=CANCELADA, restore estado=PLANIFICADA.
- Rutas navegacion: DELETE activo=false; no existe restore.

GeoJSON:
- Usar Body raw JSON en Postman.
- Evitar scripts/herramientas que serialicen mal arrays anidados.

Estado de creacion de objetos:
- Revisar estado_operacion_creacion en respuestas.
- Si es PLANIFICADA, el objeto fue creado mientras la operacion estaba en planeacion.
- Si es ACTIVA, el objeto fue creado mientras la operacion estaba activa.
- Este campo queda fijo aunque despues cambie operacion.estado.
- Para datos creados antes del parche, el valor fue rellenado con el estado actual de la operacion al momento de aplicar la migracion.

Enums canonicos:
- Operacion: PLANIFICADA, ACTIVA, CERRADA, CANCELADA.
- Prioridad: BAJA, MEDIA, ALTA.
- Personal rol: CUT, CET, CELL.
- Vehiculo estado: DISPONIBLE, ASIGNADO, MANTENIMIENTO, BAJA.
- Equipo categoria: COMUNICACION, TACTICO.
- Creador: USUARIO, PERSONAL.
- Chat tipo_mensaje: NORMAL, URGENTE, SISTEMA.
- Aviso tipo_aviso: NOVEDAD, CONTACTO, EMERGENCIA, INFORMATIVO.
- Aviso estado: RECIBIDO, ATENDIDO.
- Ruta tactica estado: PLANIFICADA, ACTIVA, COMPLETADA, CANCELADA.
